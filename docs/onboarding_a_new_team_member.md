# Onboarding a New Team Member to a Workbench

A new team member should not begin by editing the repo.

They should begin by understanding the Workbench promise:

```text
Pack → Reason → Capture Back → Verify → Commit + Repack
```

The first onboarding goal is orientation, not contribution.

## Newcomer first-read packet

Start with:

```text
README.md
SPINE.md
LLM_READ_FIRST.md
MAP.md
docs/first_use.md
docs/what_to_upload_to_an_llm.md
```

## First useful action

Ask the newcomer to write a first-read reaction note.

The note is valuable because it reveals whether the Workbench is legible to someone who was not present for its creation.

## Suggested note path

```text
notes/<name_or_role>_first_read_reaction.md
```

## Good signs

A well-onboarded newcomer can say:

```text
This is a repo-backed way to preserve reasoning, decisions, and context across AI-assisted work.
I do not need to understand every file first.
I need to understand the loop, the authority, and how useful reasoning gets captured back.
```

## What to avoid

- Do not start with broad refactoring.
- Do not ask the newcomer to judge every file.
- Do not make the first action a code change.
- Do not confuse read-only orientation with ownership.
